# AI 编程面试平台分阶段实施计划

**目标：** 在现有“题目 / 候选人 / 面试 / OpenCode 环境 / 聊天记录 / AI 评估基础版”能力之上，按依赖关系补齐企业级 AI 编程面试平台，优先打通最小业务闭环：登录 -> 选题 -> 创建面试 -> 生成候选人链接 -> 候选人作答并提交 -> AI 初评 -> 面试官复核。

**规划原则：**
- 先补基础对象、权限、状态机，再做页面和流程。
- 优先打通单候选人 MVP 闭环，再补批量能力、推荐、监测和分析。
- 题目模板、评估标准、面试快照必须先于 AI 评估与复核。
- 候选人主链路必须先于实时监测、过程回放、智能推荐等增强项。

---

## 1. 当前已完成能力基线

结合当前仓库实现，已经具备以下基础能力：

- 后台已有基础的题目管理、候选人管理、面试管理页面与 CRUD。
- 已支持按候选人 + 题目创建面试，并生成候选人访问链接。
- 候选人可通过 token 进入页面并启动 OpenCode 编程环境。
- 面试结束后已有 AI 评估服务、聊天记录读取能力和基础评估结果展示。
- 面试列表已有搜索、状态筛选、复制链接、查看聊天记录、查看 AI 评估结果。

这意味着本次规划不是从零设计，而是要把当前“基础原型”演进为“有角色权限、可复核、可运营”的正式流程平台。

---

## 2. 项目目标总结

### 2.1 MVP 核心闭环

MVP 必须优先打通以下链路：

1. 企业管理员 / 面试官登录并进入自己的工作台。
2. 选择系统题目模板或企业题目模板。
3. 创建单场面试任务并分配面试官。
4. 生成候选人专属链接。
5. 候选人进入启动页，确认规则后开始作答。
6. 候选人提交，系统结束面试并触发 AI 初评。
7. 面试官进入复核页，查看 AI 初评、过程记录和评语，给出最终结论。

### 2.2 非 MVP 但应在第二阶段补齐

- 批量导入候选人、批量创建任务、保存草稿、导出结果。
- 题目模板私有 / 企业共享能力、标签筛选、题目预览。
- 题目类型驱动的 AI 评估策略。
- 面试详情页的过程记录、异常提示、手动结束面试。

### 2.3 后续增强项

- 智能题目推荐。
- 实时屏幕、代码轨迹、操作日志回放。
- 收藏、筛选历史、批量导入题目。
- 更细颗粒度的题目权限（部门 / 指定人员）。
- 高级分析报表与复盘能力。

## 3. 核心对象与关键规则抽象

## 3.1 核心业务对象

### A. 企业管理员
- 作用：企业内全局管理者。
- 能力：查看全部面试、创建任务、管理企业共享题目、导出结果、终审。
- 前置性：必须先定义其权限边界，否则后续页面权限无法落地。

### B. 面试官
- 作用：执行面试与复核的业务角色。
- 能力：查看自己负责的面试、创建自己负责的面试、查看 AI 初评、提交人工复核。
- 前置性：必须先定义数据范围规则，否则列表页、详情页、复核页无法判断可见数据。

### C. 候选人
- 作用：通过专属链接参加面试。
- 规则：无需注册，不属于后台账号体系；只能通过 token 访问对应面试；只允许进入当前面试。

### D. 题目模板
- 建议由当前 `Problem` 演进而来。
- 必备字段：标题、岗位、难度、语言、题目类型、标签、项目背景、任务目标、验收点、工作区模板、评估标准、可见范围、创建人、所属企业。
- 状态 / 类型：`system_builtin / enterprise_shared / private`。
- 关键规则：
  - 题目类型是系统内置枚举，不允许企业自行新增或编辑。
  - 模板内容可修改，但创建面试后必须生成“题目快照”，避免后续改题影响历史结果。

### E. 题目类型
- 建议独立为系统常量或配置表。
- 示例：算法题、系统设计题、代码实现题、调试题、SQL 题。
- 关键规则：题目类型直接决定候选人作答容器要求和 AI 评估策略，不应依赖页面临时判断。

### F. 评估标准
- 作用：描述 AI 初评和人工复核应参考的验收标准。
- 结构建议：验收点清单 + 评分维度 + 权重 + 评语提示。
- 关键规则：评估标准跟随题目模板版本化，并在创建面试时快照到面试记录。

### G. 面试任务
- 建议继续使用当前 `Interview` 作为一名候选人的一条面试记录。
- 必备字段：候选人信息快照、面试官、题目模板快照、时长、候选人 token、状态、开始 / 结束时间、提交时间、结束原因、OpenCode 运行信息。
- 关键规则：
  - 一条面试只对应一名候选人。
  - 候选人开始后才进入正式状态机。
  - “保存草稿”不要直接塞进正式面试状态，建议独立 `InterviewDraft` 或 `InterviewBatchDraft`。

### H. AI 初评结果
- 作用：机器给出的首轮评估结果。
- 必备字段：状态、总分、分维度得分、摘要、证据、原始结果、失败原因、触发时间。
- 关键规则：
  - 必须与具体面试记录绑定。
  - 必须基于面试快照 + 题目类型 + 评估标准执行。
  - 可重试，但需要保留最近一次结果和重试信息。

### I. 人工复核结论
- 作用：最终业务判定。
- 必备字段：结论、评语、复核人、复核时间、是否终审。
- 状态：`通过 / 不通过 / 待定`。
- 关键规则：
  - AI 结果只能辅助，不能替代人工终结论。
  - 已有终审结论后，重新触发 AI 初评不能直接覆盖最终人工结论。

### J. 过程记录 / 事件流
- 作用：支撑详情页、异常提示、复核依据、后续过程回放。
- 建议记录：开始、打开环境、心跳、提交、超时、手动结束、AI 初评触发、AI 初评完成 / 失败。
- 前置性：详情页和复核页都依赖该对象，应该在 MVP 阶段同步落地。

## 3.2 对象关系

- 企业管理员 / 面试官 属于同一企业账号体系，面向后台登录。
- 题目模板属于系统或企业；企业模板可分为私有和企业共享。
- 面试任务关联一个候选人、一个面试官、一个题目模板快照。
- AI 初评结果附着在面试任务上。
- 人工复核结论附着在面试任务上，并引用 AI 初评结果作为依据。
- 过程记录附着在面试任务上，为详情页和复核页提供证据。

## 3.3 必须前置固化的状态机

### 面试任务状态机

建议尽快从当前三态升级为：

- `pending`：已创建，待候选人开始。
- `in_progress`：候选人作答中。
- `ai_evaluating`：候选人已提交或被结束，系统正在初评。
- `pending_review`：AI 初评已可查看，等待人工复核。
- `completed`：人工复核已完成。

建议补充字段而不是新增过多业务状态：
- `endReason`：`submitted / timeout / interviewer_ended / system_error`
- `aiEvaluationStatus`：`pending / queued / running / completed / failed`

### 页面按钮与状态关系

- 面试列表页：
  - `复制链接`：`pending / in_progress`
  - `查看详情`：全部状态
  - `进入复核`：`pending_review / completed`
  - `导出结果`：至少要求有 AI 初评结果，最佳为人工复核完成后
- 候选人启动页：仅 `pending`
- 候选人作答页：仅 `in_progress`
- 复核页：至少要求 AI 初评已完成或失败可见

## 3.4 必须前期固化的规则

- 题目模板创建后，面试必须存快照，不能直接引用“活模板”做评估。
- 题目类型不可编辑，评估策略由系统内置。
- 后台所有查询都必须先经过角色数据权限过滤。
- 候选人 token 只允许访问单一面试，面试完成后只展示完成态。
- AI 初评失败不能阻塞进入人工复核，只是复核页需显示失败原因。
- 保存草稿、批量导入、批量发送属于“任务编排层”，不要污染面试主状态机。

---

## 4. 推荐阶段顺序总览

推荐顺序：

`Phase 1 基础域模型与权限骨架`
-> `Phase 2 MVP 题目模板与单场任务创建`
-> `Phase 3 MVP 候选人作答与面试执行`
-> `Phase 4 MVP AI 初评与人工复核闭环`
-> `Phase 5 第二阶段补齐`
-> `Phase 6 后续增强`

阶段依赖说明：

- Phase 2 依赖 Phase 1 的角色体系、状态机、模板模型和数据权限。
- Phase 3 依赖 Phase 2 已能创建正式面试任务并生成候选人链接。
- Phase 4 依赖 Phase 3 已有候选人提交结果、过程记录和结束事件。
- Phase 5 依赖 Phase 4 已有稳定的单场闭环，才能做批量、导出和更多运营能力。
- Phase 6 依赖 Phase 5 已沉淀标签、过程流、评估结果等数据，才能做推荐、监测和高级分析。

## 5. 分阶段实施计划

## Phase 1：基础域模型与权限骨架

**目标：** 把当前“原型式后台”升级为可承载企业级流程的基础骨架。

**本阶段重点：**
- 建立后台登录页。
- 建立用户、企业、角色、数据权限模型。
- 升级题目模板、面试任务、AI 初评、人工复核的数据结构。
- 升级面试状态机。

**涉及页面：**
- 登录页
- 管理后台公共布局
- 面试列表页（先接入角色权限和新状态展示）

**页面动作：**
- 企业管理员 / 面试官登录
- 登录后根据角色看到自己的默认首页
- 面试列表按角色过滤数据

**后端 / 数据前置：**
- 新增账号与角色对象；建议同时引入 `Organization` 作为企业隔离边界。
- `Problem` 演进为题目模板模型，至少增加：题目类型、可见范围、所属企业、创建者、标签、岗位、难度。
- `Interview` 增加：面试官、模板快照、评估标准快照、最终结论、结束原因、新状态机字段。
- 新增人工复核对象或在面试记录中增加 review 子结构。

**完成标志：**
- 能登录。
- 能区分企业管理员和面试官的数据范围。
- 后续页面开发不再依赖旧三态和裸 `Problem` 结构。

**为什么必须先做：**
- 没有角色和数据权限，面试列表、复核页、题库共享都无法正确实现。
- 没有模板快照和评估标准快照，AI 初评与历史结果无法稳定落地。

## Phase 2：MVP 题目模板与单场任务创建

**目标：** 打通从“选题”到“创建单场面试”的后台准备链路。

**涉及页面：**
- 题目库页
- 新建面试页
- 面试列表页

**本阶段实现内容：**
- 题目库页支持系统题目、企业共享题目、私有题目三类视图。
- 支持题目新建、编辑、可见范围切换、预览。
- 支持基础筛选：关键词、题型、难度、岗位、语言、标签。
- 新建面试页支持单候选人创建、选择题目模板、指定面试官、设置时长、生成候选人链接。
- 创建时写入题目快照和评估标准快照。

**本阶段先不做：**
- 智能推荐
- 收藏和历史筛选
- Markdown / JSON 批量导入题目
- 批量候选人创建

**完成标志：**
- 面试官可以在权限范围内选题并创建正式面试任务。
- 面试列表能看到待开始面试，并复制候选人链接。

**依赖：**
- 依赖 Phase 1 的模板模型、角色模型、状态机。

## Phase 3：MVP 候选人作答与面试执行

**目标：** 打通候选人真正参加面试的主链路。

**涉及页面：**
- 候选人启动页
- 候选人作答页
- 面试详情页
- 面试列表页

**本阶段实现内容：**
- 候选人启动页展示公司名、岗位、规则说明、开始确认。
- 候选人点击开始后进入作答页，显示项目背景、任务目标、验收点、计时器、打开 OpenCode 环境入口。
- 候选人支持“结束并提交”。
- 面试详情页支持查看候选人状态、剩余时间、过程记录、异常提示、复制面试链接、手动结束面试。
- 系统记录关键事件流，作为复核依据。

**本阶段建议保留的最小技术策略：**
- 自动保存、离线提示先做轻量提示，不在本阶段深做离线恢复。
- 实时屏幕同步和代码轨迹回放先不做，只先记录可落库的过程事件。

**完成标志：**
- 候选人可以独立完成一次面试并显式提交。
- 面试官可以从详情页判断面试是否正常结束，并查看基本过程记录。

**依赖：**
- 依赖 Phase 2 已有正式面试任务、候选人 token 和模板快照。

## Phase 4：MVP AI 初评与人工复核闭环

**目标：** 完成业务最小闭环，让“结果可判定、可留痕、可复查”。

**涉及页面：**
- 面试详情页
- 复核页
- 面试列表页

**本阶段实现内容：**
- 候选人提交或面试官结束面试后，状态进入 `ai_evaluating`。
- AI 初评完成后，状态切到 `pending_review`。
- 复核页展示：AI 初评分、维度说明、摘要、聊天历史、过程记录摘要、人工评语区。
- 支持人工结论：通过 / 不通过 / 待定。
- 支持保存评语、重新触发 AI 初评。
- 面试列表展示 AI 初评分和人工最终结论。

**本阶段必须做的评估升级：**
- 从当前“通用聊天历史评分”升级为“题目类型 + 题目快照 + 评估标准”的评估入口。
- 即便第一版内部仍复用现有聊天记录数据，也必须先把评估接口抽象成按题型分发的策略层。

**完成标志：**
- 单场面试从创建到最终人工结论可闭环。
- AI 初评结果可重试、可查看、可供人工复核。

**依赖：**
- 依赖 Phase 3 已有提交事件、结束事件和过程记录。

## Phase 5：第二阶段补齐

**目标：** 补足后台运营效率和题库运营能力，让系统可用于真实招聘批量场景。

**涉及页面：**
- 新建面试页
- 面试列表页
- 题目库页
- 复核页

**本阶段实现内容：**
- 批量粘贴导入候选人、批量创建面试任务。
- 保存草稿 / 草稿恢复。
- 创建并发送链接。
- 面试结果导出。
- 题目预览增强。
- 企业共享 / 私有切换完善。
- 复核页导出报告。

**建议放在本阶段的原因：**
- 这些能力会放大前面主流程的价值，但不是单场面试闭环的前置条件。

**完成标志：**
- 面试官可高效安排多名候选人的面试并批量处理结果。

**依赖：**
- 依赖 Phase 4 已稳定支持单场完整闭环，否则批量功能只会放大流程问题。

## Phase 6：后续增强

**目标：** 在稳定流程之上增加智能化和差异化能力。

**建议纳入内容：**
- 基于 JD / 简历的题目智能推荐与一键采纳。
- 实时屏幕、代码轨迹、操作日志、过程回放。
- 收藏题目、历史筛选、批量导入题目。
- 更细粒度的题目权限（部门 / 指定人员）。
- AI 阶段性测评、异常风险提示、高级分析报表。

**依赖：**
- 依赖前面阶段已沉淀模板标签、过程事件、AI 初评结果、人工结论等结构化数据。

## 6. MVP 必需 / 第二阶段补齐 / 后续增强划分

### MVP 必需
- 登录与角色权限
- 题目模板基础模型和可见范围
- 单场面试创建
- 候选人启动 / 作答 / 提交

### 第二阶段补齐
- 批量候选人导入
- 批量创建任务
- 保存草稿
- 导出结果 / 导出报告
- 题库筛选完善
- 企业共享题库运营能力

### 后续增强
- 智能推荐
- 实时监测与回放
- 收藏 / 筛选历史
- 题目批量导入
- 高级分析与风险识别

## 7. 对 Codex 的执行建议

- 每个 Phase 先做“数据模型 + API + 页面骨架”，再做交互细节。
- 先统一状态机和权限判断，再改页面按钮，否则很容易出现页面先成型、后端状态不支撑的返工。
- 每个 Phase 完成后都要做一次端到端验证，而不是只看单页 UI。
- AI 评估相关能力不要继续直接耦合到当前单一实现，应尽快抽象成可按题型扩展的评估策略接口。

## 8. 需补充确认但不阻塞整体规划的问题

1. 登录体系是否接入现有企业账号系统，还是先做平台内置账号密码登录？
2. 当前是否只考虑单企业环境，还是要为后续多企业隔离预留 `Organization` 模型？
3. 面试官是否可以给出最终结论，还是企业管理员才有终审权限？
4. “保存草稿”是保存单场创建表单，还是要支持批量面试任务草稿？
5. 复核页中的“重新触发 AI 初评”是否要求保留历史版本，还是只保留最近一次结果？
6. 导出结果的格式优先级是什么：CSV、Excel、PDF 报告？
7. 候选人提交后是否允许再次进入作答页查看内容，还是只显示完成态？
8. 题目模板的“评估标准”是否需要结构化表单，还是先支持文本 + 验收点列表即可？
